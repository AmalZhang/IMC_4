# IMC 双人协作说明

这份文档给你和队友使用，目标是把协作成本降到最低，尤其适合其中一人不熟悉 Git 的情况。

## 角色分工

- 主线维护者：建议由你担任。
  负责主分支、合并代码、平台上传、确认 `active` 版本。
- 实验协作者：建议由队友担任。
  负责独立尝试某一个方向，不直接覆盖主线。

无论谁写代码，都不要两个人同时直接改同一个主文件后再手工拼接。

## 当前仓库约定

- 提交文件：`trader_round3.py`
- 本地分析文件：`round3_analysis.py`
- 规则索引：`AGENTS.md`
- 比赛协作流程：`IMC协作提交流程.md`
- 本协作文档：`COLLAB.md`

## 最稳的协作方式

采用“一人控主线，一人做实验”的方式：

1. 主线永远保持可提交。
2. 每个实验都在单独分支完成。
3. 只有主线维护者负责上传 IMC。
4. 平台结果统一反馈回主线维护者和 Codex。

## 给不熟悉 Git 的队友的最简流程

队友只需要学会下面 6 个动作。

### 1. 第一次拿仓库

如果你已经把仓库同步到了 GitHub 或其他远端，队友执行：

```powershell
git clone <仓库地址>
cd IMC
```

### 2. 开始一个新任务前，先更新主线

```powershell
git checkout main
git pull
```

### 3. 为这次改动新建一个分支

分支名只写一件事，不要混很多目标：

```powershell
git checkout -b feat/r3-option-smile
```

分支名示例：

- `feat/r3-option-smile`
- `feat/r3-delta-hedge`
- `fix/r3-position-cap`
- `fix/r3-timeout`

### 4. 改代码后先做本地检查

```powershell
python -m py_compile trader_round3.py round3_analysis.py
```

如果这一步不过，不要提交。

### 5. 提交自己的改动

```powershell
git status
git add trader_round3.py round3_analysis.py
git commit -m "feat: improve round 3 option pricing"
git push -u origin feat/r3-option-smile
```

### 6. 不要直接改 main

队友不要直接在 `main` 上写代码、提交、推送。  
所有实验都先走分支。

## 推荐协作节奏

### 方案 A：你们两个人都写代码

- 你维护 `main`
- 队友在分支上实验
- 队友完成后把修改说明发你
- 你负责审查并合入 `main`
- 你负责上传到 IMC 平台

### 方案 B：队友主要用 ClaudeCode / Agent 帮忙

- 队友先拉最新 `main`
- 队友在自己分支里让 ClaudeCode 改代码
- ClaudeCode 只处理一个问题，不做大杂烩
- 队友把分支推上远端
- 你拉下来审查，再决定是否合并

这比两边都让 Agent 直接改主线稳得多。

## 每次改动必须附带的信息

无论是你还是队友，提交代码时都要说明 4 件事：

1. 改了什么
2. 为什么改
3. 预期改善哪个问题
4. 本地检查是否通过

建议用下面模板发消息：

```text
分支名：feat/r3-option-smile
改动内容：调整 VEV 公平价和做市边界
目标：减少 5300/5400 连续亏损
本地检查：py_compile 已通过
风险：可能会降低成交量
```

## 哪些东西不要同时改

以下区域冲突概率最高，最好一次只由一个人维护：

- `Trader.run`
- 统一下单入口
- 仓位裁剪逻辑
- `traderData` 结构
- 期权对冲和现货对冲的联动

如果你已经在改这些区域，队友就不要并行改同一块。

## 适合拆给队友的任务

比较适合并行的方向：

- 单独分析 `HYDROGEL_PACK`
- 单独分析某几个 `VEV_*` 的公平价
- 做日志清理、超时优化
- 写分析脚本，不动主交易逻辑
- 做手动挑战 EV 计算

不适合拆开的方向：

- 同时重写同一个 `Trader.run`
- 同时修改统一下单函数和仓位限制函数

## 平台提交规则

- 只有一个人负责上传到 IMC。
- 只有一个版本应该被设为 `active`。
- 平台结果出来后，不要靠口头描述，尽量贴：
  - 总 PnL
  - 分产品表现
  - debug log
  - 是否超时 / 是否报错

## 给队友的实际建议

如果队友不熟悉 Git，最适合他的做法不是学很多 Git 命令，而是先记住这几条：

1. 改代码前先 `git pull`
2. 不在 `main` 上直接改
3. 一次分支只做一件事
4. 提交前先 `py_compile`
5. 提交信息写清楚“改了什么”
6. 不确定就不要自己合并，让主线维护者处理

## 如果不想用 GitHub，也可以这样同步

如果你们暂时不想配 GitHub，也可以用下面方式：

- 方案 1：局域网/网盘同步整个仓库目录
- 方案 2：用压缩包同步分支快照
- 方案 3：OneDrive/坚果云/Dropbox 同步仓库目录

但这些方案都有明显风险：

- 容易互相覆盖
- 难回滚
- 难比较改了什么

所以优先建议 Git + 远端仓库。

## 一句话原则

让一个人控制主线和比赛平台提交，另一个人在分支里安全试验。  
不会 Git 没关系，但不要跳过分支、检查和说明这三步。
